# Usability Survey Generation Brief

## Purpose

Use this file with Google Forms, Gemini, or another Google AI tool to generate the usability survey required for the HCI assignment.

Assignment requirement covered:

- Part B: Conduct a usability survey.
- Target potential users.
- At least 10 participants recommended.
- Include closed-ended rating questions and open-ended questions.
- Focus on ease of use, clarity, navigation, aesthetics, and functionality.
- Show the low-fidelity prototype to participants before they answer.
- Analyze results with graphs such as bar charts and pie charts.

## Survey Context

Project name: **AuraBeat**

Project description:

AuraBeat is an AI-powered music generation web application. Users can register, generate music using prompts, choose music styles, upload reference audio, manage generated tracks in a library, play tracks through a persistent audio player, edit track metadata, use a basic music editor, manage their profile, and create API keys for developer access.

Prototype being evaluated:

Low-fidelity digital wireframe prototype created in Figma. It includes login/register, dashboard, create music, advanced upload options, generation result, library grid/list, edit/delete modals, persistent audio player, editor, profile, and API platform.

Survey audience:

- Students.
- Content creators.
- Beginner music creators.
- Web app users.
- Potential users of AI tools.
- Optional: software/developer users for API platform feedback.

Recommended participant count:

- Minimum: 10 participants.
- Ideal for better charts: 15 to 20 participants.

## Instructions For Participants

Before answering the survey, participants should be shown the low-fidelity prototype and asked to imagine completing these tasks:

1. Create a new account or log in.
2. Reach the dashboard.
3. Start creating an AI music track.
4. Locate advanced options and upload reference audio.
5. Understand the generation result screen.
6. Find the generated track in the Library.
7. Play the track.
8. Edit the track title/tags.
9. Delete a track safely.
10. Find profile and API platform options.

## Task Completion Questions

Use this scale:

- 1 = Very difficult.
- 2 = Difficult.
- 3 = Neutral.
- 4 = Easy.
- 5 = Very easy.

Questions:

1. How easy was it to understand how to log in or create an account?
2. How easy was it to identify what to do first on the dashboard?
3. How easy was it to find the Create Music feature?
4. How easy was it to understand where to type the music prompt?
5. How easy was it to understand the style/genre selection?
6. How easy was it to find advanced options such as lyrics and reference audio upload?
7. How easy was it to understand the generation result screen?
8. How easy was it to find a generated track in the Library?
9. How easy was it to identify the Play action?
10. How easy was it to identify Edit and Delete actions?
11. How easy was it to find Profile settings?
12. How easy was it to find API key management?

## Likert-Scale Usability Questions

Use this scale:

- 1 = Strongly disagree.
- 2 = Disagree.
- 3 = Neutral.
- 4 = Agree.
- 5 = Strongly agree.

Questions:

1. The prototype's navigation was clear.
2. The screen layout was easy to understand.
3. The labels and buttons used familiar language.
4. The Create Music flow seemed simple enough for a first-time user.
5. The advanced options were useful and not too confusing.
6. The Library screen made it easy to search and manage tracks.
7. The audio player placement was easy to notice.
8. The edit and delete actions were easy to understand.
9. The prototype included the features I would expect from an AI music app.
10. The system provided enough feedback after generation actions.
11. I felt confident that I could complete the main tasks.
12. The mobile navigation concept seemed understandable.
13. The overall structure of the app was logical.
14. I would be interested in using a polished version of this application.

## Optional SUS-Style Questions

Use the same 1 to 5 agreement scale.

1. I think I would like to use AuraBeat frequently.
2. I found the prototype unnecessarily complex.
3. I thought the prototype was easy to use.
4. I think I would need help before using this app.
5. I found the app sections well integrated.
6. I thought there was too much inconsistency in the prototype.
7. I imagine most users would learn this app quickly.
8. I found the prototype confusing to navigate.
9. I felt confident using the prototype.
10. I needed to learn many things before I could get started.

For analysis, reverse-score negative items 2, 4, 6, 8, and 10 if calculating a SUS-like score.

## Open-Ended Questions

Use these questions:

1. What part of the prototype was easiest to understand?
2. What part of the prototype was most confusing?
3. Which screen needs the most improvement and why?
4. Were any important features missing from the prototype?
5. What would make the Create Music flow easier?
6. What would make the Library easier to use?
7. Did the prototype clearly show how audio playback works?
8. What changes should be made before creating the high-fidelity prototype?
9. Any additional comments or suggestions?

## Data Analysis Plan

Create these charts after collecting responses:

1. Pie chart: Participant roles.
2. Pie chart: AI tool experience.
3. Bar chart: Ease of completing each task.
4. Bar chart: Average Likert score by usability category.
5. Stacked bar chart: Agreement distribution for navigation and layout questions.
6. Bar chart: Most confusing screens/features based on coded open-ended responses.
7. Optional: SUS-like average score.

## Expected Feedback Themes To Watch For

When analyzing open-ended responses, look for themes such as:

- Users want Create Music to be more step-by-step.
- Users may not immediately understand Gold/credits.
- Advanced options may feel crowded.
- Upload controls may need clearer labels.
- Library actions need visible icons and confirmations.
- Users may want clearer generation progress.
- Users may want search, filter, and sorting in Library.
- Mobile navigation must remain simple.
- API Platform may be confusing for non-developers.

These themes should directly inform the high-fidelity prototype.
