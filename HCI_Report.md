# 1. Introduction

This report presents the prototype development and usability evaluation process for **AuraBeat**, an AI-powered music generation platform. The assignment required creating a low-fidelity prototype, conducting a usability survey, analyzing user feedback, developing a high-fidelity prototype based on that feedback, and presenting the complete iteration process in a final report.

The purpose of this work is to demonstrate the importance of user-centered design and iterative improvement in Human Computer Interaction. Instead of directly building a polished interface, the project begins with a low-fidelity prototype so that usability problems can be identified early. Feedback from potential users is then used to improve the high-fidelity prototype.

---

# 2. Project Overview

## 2.1 Project Name

**AuraBeat - AI-Powered Music Generation Platform**

## 2.2 Project Description

AuraBeat is a web-based AI music generation platform. It allows users to create an account, generate music tracks using prompts, choose music styles, upload reference audio, manage generated tracks in a library, play tracks through a persistent audio player, edit track metadata, access a basic music editor, manage their profile, and create API keys for developer access.

## 2.3 Project Goals

The main goals of AuraBeat are:

1. To make AI music generation accessible through a simple web interface.
2. To allow users to generate music using natural language prompts.
3. To provide a personal track library for saved music.
4. To support playback and basic track management.
5. To provide a clear and usable interface for both beginner users and advanced/developer users.
6. To demonstrate a complete FYP concept through HCI prototyping.

## 2.4 Target Users

| User Group | Description |
|---|---|
| Students | Users who need background music for projects or presentations |
| Content creators | Users who need quick music ideas for videos or social media |
| Beginner music creators | Users who want to experiment with AI-generated music |
| Developers | Users who may want API access to tracks and generation |
| Admin users | Users who manage platform data during demonstration |

# 4. Methodology

The project followed an iterative HCI design process:

1. Identify the project goals and target users.
2. Create a low-fidelity prototype to represent major screens and flows.
3. Show the low-fidelity prototype to potential users.
4. Collect usability feedback through a structured survey.
5. Analyze quantitative and qualitative responses.
6. Identify common usability issues.
7. Redesign the prototype into a high-fidelity version.
8. Document the changes and learning outcomes.

This method supports user-centered design because the final interface is influenced by participant feedback rather than only developer assumptions.

## 6.4 Survey Questionnaire

The questionnaire included closed-ended questions, Likert-scale ratings, and open-ended feedback.

### Demographic Questions

1. What is your age range?
2. What is your current role?
3. How often do you use web applications?
4. Have you used any AI-based tool before?
5. Have you used any music/audio creation tool before?

### Task-Based Questions

Participants rated task ease from 1 to 5:

1. Understanding how to log in or create an account.
2. Identifying what to do first on the dashboard.
3. Finding the Create Music feature.
4. Understanding where to enter a music prompt.
5. Understanding style/genre selection.
6. Finding advanced options and reference audio upload.
7. Understanding the generation result screen.
8. Finding a generated track in the Library.
9. Identifying the Play action.
10. Identifying Edit and Delete actions.
11. Finding Profile settings.
12. Finding API key management.

### Likert-Scale Questions

Participants rated agreement from 1 to 5:

1. The prototype navigation was clear.
2. The screen layout was easy to understand.
3. Labels and buttons used familiar language.
4. The Create Music flow seemed simple enough for first-time users.
5. Advanced options were useful and not too confusing.
6. The Library screen made it easy to search and manage tracks.
7. The audio player placement was easy to notice.
8. Edit and delete actions were easy to understand.
9. The prototype included expected features for an AI music app.
10. The system provided enough feedback after generation actions.
11. I felt confident completing the main tasks.
12. The mobile navigation concept was understandable.
13. The overall structure of the app was logical.
14. I would be interested in using a polished version of this application.

### Open-Ended Questions

1. What part of the prototype was easiest to understand?
2. What part was most confusing?
3. Which screen needs the most improvement?
4. Were any important features missing?
5. What would make the Create Music flow easier?
6. What would make the Library easier to use?
7. Did the prototype clearly show how audio playback works?

# 8. Changes Implemented After Feedback

The survey feedback was used to improve the high-fidelity prototype. The major changes are summarized below.

# 11. Conclusion

This assignment demonstrated the importance of iterative design in Human Computer Interaction. The low-fidelity prototype helped identify early usability concerns before visual polish was added. The usability survey provided structured feedback from potential users, including both rating-based and open-ended responses. Based on the feedback, the high-fidelity prototype improved navigation, create-flow clarity, generation feedback, library management, error prevention, mobile usability, and overall visual presentation.

Through this process, it became clear that user feedback is essential for designing usable systems. A feature may be technically complete but still confusing if labels, navigation, feedback, and layout are not designed around user expectations. The final high-fidelity AuraBeat prototype better represents a user-centered AI music generation platform.
